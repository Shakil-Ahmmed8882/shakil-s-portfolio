# cmd-responsive — Figma-Aligned Responsive System Principles

Responsiveness does not grant freedom to redesign.

Figma is the source of truth.  
All responsive behavior must preserve the intent, structure, and decisions defined in Figma.

---

## 1. FIGMA AS ABSOLUTE REFERENCE

- Desktop layout in Figma is the foundation, not a suggestion
- No structural deviation is allowed at base (desktop) level
- Spacing, alignment, proportions, and hierarchy must match exactly
- Do not reinterpret, redesign, or “improve” Figma decisions

---

## 2. DERIVATION, NOT CREATION

- Smaller layouts must be derived from Figma, not invented
- Responsiveness is adaptation, not redesign
- Every change must feel like a natural extension of the original layout
- If a layout decision cannot be traced back to Figma intent, it is invalid

---

## 3. HIERARCHY PRESERVATION

- Visual importance defined in Figma must remain unchanged
- Primary, secondary, and tertiary elements must retain their roles
- Compression must not flatten hierarchy
- Emphasis must remain consistent across all sizes

---

## 4. SPACING FIDELITY

- Spacing relationships from Figma must be preserved proportionally
- Do not introduce arbitrary spacing systems
- Density changes must feel intentional, not improvised
- Visual rhythm must stay consistent with the original design

---

## 5. ALIGNMENT LOYALTY

- Alignment rules in Figma must remain intact
- Elements must not drift or re-align arbitrarily
- Layout shifts must still respect original alignment logic
- Misalignment is considered a structural failure

---

## 6. COMPONENT INTEGRITY

- Components must retain their identity across breakpoints
- Do not fragment or merge components without clear structural reasoning
- Each component must behave as a predictable unit
- Internal composition must stay consistent with Figma patterns

---

## 7. CONTROLLED ADAPTATION

- Adaptation must reduce complexity, not distort meaning
- Non-critical elements may be deferred, never critical ones
- Layout changes must feel inevitable, not experimental
- Every adaptation must preserve usability first

---

## 8. TYPOGRAPHY CONSISTENCY

- Typography scale must respect Figma hierarchy
- Text behavior must not break layout or readability
- Line structure and readability must remain controlled
- Do not introduce new typography patterns outside Figma system

---

## 9. CONTENT RESILIENCE

- Content must fit within the intended design constraints
- Overflow must be handled without breaking structure
- Dynamic data must not distort layout logic
- Empty and edge states must still reflect Figma intent

---

## 10. NO UNAUTHORIZED VARIATIONS

- Do not introduce new layouts, patterns, or structures
- Do not improvise UI decisions outside defined design language
- Avoid “creative fixes” that diverge from the system
- Consistency across screens is mandatory

---

## 11. PREDICTABLE RESPONSIVE BEHAVIOR

- Similar components must adapt in similar ways
- Users must not experience inconsistent transformations
- Interaction patterns must remain stable across sizes
- Behavior must feel systematic, not case-by-case

---

## 12. PERFORMANCE WITHIN DESIGN INTENT

- Performance optimizations must not alter UI behavior
- Rendering efficiency must support, not compromise, design fidelity
- Responsiveness includes both visual and interaction performance

---

## 13. EDGE CASE ALIGNMENT

- Extreme cases must still feel part of the same system
- UI must not break outside ideal conditions
- All variations must still reflect the original design logic

---

## 14. DESIGN SYSTEM ENFORCEMENT

- Figma defines the design system boundaries
- All implementations must stay within those boundaries
- Reuse existing patterns instead of creating new ones
- Consistency is enforced, not optional

---

## FINAL PRINCIPLE

You are not designing.

You are executing a system defined in Figma.

Responsiveness is the discipline of preserving that system under constraint  
without distortion, deviation, or invention.
